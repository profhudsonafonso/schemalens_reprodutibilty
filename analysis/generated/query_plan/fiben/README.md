# FIBEN MongoDB Query-Plan Results

This folder contains the consolidated MongoDB query-plan validation results for the FIBEN dataset across three scale factors: SF1, SF10, and SF30.

## Scope

The query-plan validation covers all FIBEN benchmark queries:

* Q1–Q9: read queries, successfully explained across SF1, SF10, and SF30.
* Q10: insert/update workload, consistently marked as `skipped` because it is not comparable under read-query MongoDB `executionStats`.

## Consolidated files

* `fiben_query_plan_summary_all.csv`: all candidate-level query-plan summaries.
* `fiben_query_plan_components_all.csv`: all semantic component-level explain results.
* `fiben_query_plan_status_all.csv`: raw status summaries from each run.
* `fiben_query_plan_query_scale_status.csv`: execution status by query and scale.
* `fiben_query_plan_query_scale_overview.csv`: compact overview by query and scale.
* `fiben_query_plan_best_by_estimated_bytes.csv`: best candidate by estimated examined bytes.
* `fiben_query_plan_compact_candidates.csv`: compact candidate-level table for analysis.

## Scripts

The query-plan runner used to generate the per-query MongoDB explain results is located at:

`benchmark/fiben/run_fiben_mongo_query_plan.py`

The consolidation script used to merge the per-query outputs into the files in this folder is located at:

`analysis/scripts/consolidate_fiben_query_plan_results.py`

## Validation summary

The consolidation produced 180 candidate-level summary rows:

* 60 candidate summaries for SF1.
* 60 candidate summaries for SF10.
* 60 candidate summaries for SF30.

All read queries Q1–Q9 completed successfully in all scales. Q10 produced 7 skipped rows per scale, one for each selected candidate/control configuration.

## Query-plan interpretation

The results show four main physical patterns:

1. **Point lookup**

   * Q1 is a direct root lookup using `CORPORATIONID`.
   * It uses indexed access and does not require COLLSCAN or aggregation.

2. **Indexed multi-component retrieval**

   * Q2, Q3, and Q5 use indexed components and no COLLSCAN.
   * These queries are useful for comparing how different document designs reduce or preserve dereferencing paths.

3. **Associative/global scan**

   * Q4 and Q6 include COLLSCAN in at least one component.
   * These queries are dominated by broader traversal or filtering patterns, so differences among candidate designs are smaller.

4. **Transactional aggregation**

   * Q7, Q8, and Q9 use GROUP stages.
   * Their physical behavior is dominated by `transactions` and aggregation components, so candidate designs often produce similar plans.

## Methodological note about Q10

Q10 creates a new account, holding, and buy transaction. Since it is an insert/update workload, it is not directly comparable with read-query MongoDB `executionStats`. Therefore, the query-plan runner records Q10 as `skipped` across SF1, SF10, and SF30.

## Methodological note about large-scale candidate loading

For some larger-scale runs, the query-plan runner used a parameter-focused candidate loading mode to avoid materializing unnecessarily large candidate collections when the goal was to explain the physical access path for a representative query parameter.

Therefore, the most reliable fields for query-plan comparison are:

* `n_components`
* `sum_total_docs_examined`
* `sum_total_keys_examined`
* `has_IXSCAN`
* `has_COLLSCAN`
* `has_GROUP`
* `all_stages`
* `all_index_names`
* `collections_touched`

Fields derived from collection-level size statistics, such as estimated examined bytes and average object size, should be interpreted as supporting evidence rather than as complete storage-size measurements for the full candidate configuration in those parameter-focused runs.

## Raw explain files

The consolidated CSV files are the primary versioned artifacts. Raw MongoDB explain JSON files may be generated by the runner with `--save-raw-explain`, but they are not required for the consolidated analysis and may be omitted from version control due to size.

## Reproducing the Q2 query-plan explanation table

The LaTeX table used to explain FIBEN Q2 can be regenerated from the versioned CSV files in this repository.

### Script

The table-generation script is located at:

`analysis/scripts/generate_fiben_q2_query_plan_table.py`

### Required input files

The script uses the consolidated FIBEN query-plan summaries from this folder:

`analysis/generated/query_plan/fiben/fiben_query_plan_summary_all.csv`

It also uses the FIBEN benchmark aggregate results for the three scale factors:

`analysis/fiben/benchmark_aggregate_results_fiben_sf1.csv`

`analysis/fiben/benchmark_aggregate_results_fiben_sf10.csv`

`analysis/fiben/benchmark_aggregate_results_fiben_sf30.csv`

These benchmark files provide the hot-p95 winners for each scale factor. The query-plan CSV provides the MongoDB plan stages, number of components, documents examined, average object size, and estimated examined bytes.

Raw MongoDB explain JSON files are not required to regenerate this table because the relevant summarized plan information is already included in `fiben_query_plan_summary_all.csv`.

### Command

From the repository root, run:

```bash
python analysis/scripts/generate_fiben_q2_query_plan_table.py
```

### Generated outputs

The script generates:

`analysis/generated/query_plan/fiben/fiben_q2_query_plan_table_rows.csv`

`analysis/generated/query_plan/fiben/fiben_q2_query_plan_table.tex`

The CSV file is an auditable intermediate table with the values used in the LaTeX table. The `.tex` file contains the Overleaf-ready table for the FIBEN Q2 query-plan explanation.

### Ratio definition

For each scale factor, ratios are computed relative to the hot-p95 winning configuration at the same scale.

For example:

* SF1 ratios are relative to the SF1 hot-p95 winner.
* SF10 ratios are relative to the SF10 hot-p95 winner.
* SF30 ratios are relative to the SF30 hot-p95 winner.

The generated table reports:

* document-examined ratio;
* average-object-size ratio;
* estimated-examined-bytes ratio.

### Interpretation

The table is intended to connect three layers of evidence:

1. the analytical activation matrix, which explains why configurations such as descriptor embedding, references, shared references, update-aware references, and trade-off designs are considered;
2. the MongoDB query-plan evidence, which shows the physical access pattern used by each configuration;
3. the benchmark result, which identifies the hot-p95 winner for each scale.

For FIBEN Q2, all configurations use indexed access and avoid collection scans. Therefore, the relevant differences are not caused by full scans, but by the number of indexed components, dereferencing structure, number of examined documents, and physical document size.
