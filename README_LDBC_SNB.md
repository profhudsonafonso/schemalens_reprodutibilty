# SchemaLens Artifact — LDBC SNB Case Study

This repository contains the code and documentation required to reproduce the LDBC SNB experiments used in the paper on SchemaLens: Explainable Design-Space Reduction for Workload-Aware MongoDB Schema Selection.

LDBC SNB is used in the paper as a social-network benchmark dataset. It evaluates whether SchemaLens can reduce the MongoDB design space for a workload involving persons, friendships, forums, posts, comments, tags, likes, and insert operations.

The repository supports reproduction of:

methodology execution;  
MongoDB benchmark execution;  
result analysis;  
cross-scale comparison;  
paper-support outputs.

methodology/ldbc_methodology.ipynb

This step executes the SchemaLens methodology for LDBC SNB. It derives the conceptual/workload matrix, computes the analytical variables, applies the G0–G9 activation logic, and generates the artifacts used by the benchmark.

The methodology computes:

#Step 3 — MongoDB benchmark execution

Run:

benchmark/ldbc_snb/run_ldbc_snb_mongo_benchmark.py

This script:

loads the LDBC SNB scale-factor data;
materializes MongoDB candidate configurations;
executes benchmark queries;
exports benchmark outputs.

Example command for sf0.1:

python benchmark/ldbc_snb/run_ldbc_snb_mongo_benchmark.py \
  --data-dir datasets/ldbc_snb/sf0.1 \
  --artifacts-dir analysis/ldbc_snb \
  --results-dir results/ldbc_snb/sf0_1_full_fiben_format_clean \
  --execution-plan benchmark_execution_plan.csv \
  --scale-label sf0.1 \
  --execution-db-prefix ldbc_snb_sf0_1_exec_clean \
  --mongo-host 127.0.0.1 \
  --mongo-port 27018 \
  --batch-size 5000 \
  --repetitions 10 \
  --run-phase cold hot \
  --force-rebuild-db \
  --verbose

#Troubleshooting

Missing benchmark execution plan

If the benchmark reports a missing execution plan, check that the execution plan exists in:

analysis/ldbc_snb/benchmark_execution_plan.csv

and that the benchmark command uses:

--artifacts-dir analysis/ldbc_snb

IC1–IC5 return zero documents

Check whether the benchmark runner is using semantic parameter pools and whether person_knows_person_0_0.csv maps the second person column to:

person2_id

Insert queries return zero documents

This is expected. Insert queries should be validated with:

documents_written = 1

Scale-factor results look identical

Check that these three arguments point to the correct scale:

--data-dir
--results-dir
--scale-label

Each scale factor must use a separate result directory.
